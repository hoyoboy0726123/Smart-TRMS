## 1. 簡介

### 1.1 文件目的
本文件旨在詳細定義「智慧測試報告管理系統」最小可行性產品 (MVP) 的功能範圍、技術規格與業務流程。此文件將作為開發團隊（前端、後端、AI 工程師）與設計團隊的開發依據，並協助利害關係人理解產品的核心價值與交付物。

### 1.2 產品背景
目前認證實驗室（如 ISO 認證、電子、醫療、材料實驗室）產出的測試報告大多以 PDF 格式儲存。管理人員面臨以下痛點：
1. **數據碎片化**：關鍵數值埋藏在數百頁的 PDF 中，難以快速檢索。
2. **人工登錄低效**：需手動將 PDF 數據抄寫至 Excel 進行分析，耗時且易出錯。
3. **缺乏趨勢洞察**：難以跨報告追蹤特定指標的長期波動，導致品質預警反應過慢。

### 1.3 產品願景
打造一個具備 AI 智慧辨識能力的數位化中心，讓「非結構化」的實驗室報告自動轉化為「結構化」的決策數據，成為實驗室管理與品質控管的智慧大腦。

### 1.4 MVP 目標
*   實作針對標準模板 PDF 的自動化 AI 數據擷取功能。
*   建立「人機協作 (Human-in-the-loop)」的數據覆核機制，確保 100% 數據準確性。
*   提供數位化檔案庫，支援關鍵字與多維度標籤查詢。
*   自動生成特定測試指標的趨勢折線圖。
*   自動統計多份報告的合格/不合格比例。
*   提供 AI 驅動的報告摘要功能，快速掌握結論。

## 2. 目標用戶

### 主要用戶：質量管理經理 (QA Manager)
*   **用戶特徵**：
    *   負責監督產品品質與合規性。
    *   技術能力中等，熟悉辦公室軟體與內部 ERP 流程。
    *   **需求**：需要快速掌握品質趨勢、彙整合規報告，並在出現異常時追溯源頭。

### 次要用戶：實驗室技術人員/文書處理員
*   **用戶特徵**：
    *   負責上傳報告並確保數據正確錄入系統。
    *   每天處理大量紙本或數位檔案。
    *   **需求**：減少重複性的打字工作，希望系統能自動抓取數據供其確認。

## 3. 核心問題與解決方案

*   **核心問題 1：PDF 報告內的數據無法直接用於運算與分析**
    *   **MVP 解決方案**：導入 AI/OCR 引擎，針對預定義模板自動提取「測試日期」、「測試值」、「判定結果」等關鍵欄位，存入 SQL 資料庫。
*   **核心問題 2：純 AI 辨識可能存在誤差，醫療或工業認證無法容忍錯誤**
    *   **MVP 解決方案**：設計「雙欄對照」介面，讓人工在上傳時能即時對比原始文件與辨識結果，手動修正後才正式入庫。
*   **核心問題 3：難以視覺化觀測產品品質隨時間的偏移**
    *   **MVP 解決方案**：開發趨勢看板，自動將資料庫中的結構化數值依時間軸繪製成折線圖。
*   **核心問題 4：長篇報告閱讀耗時，難以快速抓到重點**
    *   **MVP 解決方案**：整合 LLM（大語言模型）在上傳時自動生成 100-200 字的簡要結論摘要。

## 4. 產品功能 (MVP 範圍)

### 4.1 智慧上傳與 AI 辨識模組
*   **功能描述**：用戶上傳 PDF 報告，系統自動進行文字辨識與結構化分析。
*   **具體子功能**：
    *   **拖放上傳**：支援單一或批次 PDF 檔案拖放至瀏覽器。
    *   **模板匹配**：根據報告特徵自動識別屬於哪一家實驗室的標準格式。
    *   **關鍵欄位擷取**：擷取報告編號、日期、實驗室名稱、產品型號、測試結論（Pass/Fail）。
    *   **數據點提取**：精準抓取表格中的關鍵測試數值（如電壓、壓力、雜質含量）。
    *   **AI 文本摘要**：利用 LLM 摘要報告末尾的建議或結論。

### 4.2 人機協作覆核工作台
*   **功能描述**：提供一個介面讓用戶在數據永久存檔前進行最終核對。
*   **具體子功能**：
    *   **雙欄對照視窗**：左側為 PDF 檢視器（Highlight 辨識區域），右側為可編輯表單。
    *   **異常標示**：若 AI 信心程度較低，自動以紅框標示該欄位提醒人工核對。
    *   **即時修正存檔**：用戶可直接在右側表單修改數值，點擊「確認入庫」完成流程。
    *   **狀態追蹤**：標記報告狀態為「待審核」、「已確認」、「已作廢」。

### 4.3 數位檔案管理中心
*   **功能描述**：結構化儲存所有已確認的報告，提供高效管理。
*   **具體子功能**：
    *   **多維度過濾**：可依日期區間、產品類別、實驗室名稱、合格狀態篩選。
    *   **全文檢索**：支援搜尋報告編號、產品關鍵字及 AI 摘要內容。
    *   **檔案下載**：支援隨時下載原始 PDF 檔案。
    *   **邏輯歸檔**：系統自動依年月生成資料夾結構，便於瀏覽。

### 4.4 趨勢分析視覺化看板
*   **功能描述**：將數據轉化為具備洞察價值的圖表。
*   **具體子功能**：
    *   **指標趨勢圖**：用戶選擇單一指標（如：抗壓強度），系統產出隨時間變化的折線圖。
    *   **統計匯總表**：顯示該指標的平均值、最大值、最小值與標準差。
    *   **合格率圓餅圖**：統計選定範圍內「Pass」與「Fail」的比例。
    *   **比較模式**：可同時選取兩家不同實驗室的數據進行品質穩定性比較。

### 4.5 系統設定與權限管理
*   **功能描述**：確保數據安全與系統基本配置。
*   **具體子功能**：
    *   **用戶認證**：標準的帳號密碼登入機制。
    *   **角色定義**：分為「管理員（可刪除報告）」與「一般用戶（僅能上傳與查看）」。
    *   **實驗室模板配置**：允許管理員定義哪些 PDF 格式應對應哪些數據欄位（初步以硬編碼或配置檔實現）。

## 5. 用戶故事 (User Stories)

1.  **作為 QA 經理**，我想要在儀表板看到每個月的合格率趨勢，以便評估供應商的生產品質是否穩定。
2.  **作為實驗室技術員**，我想要系統自動從 50 頁的報告中抓取測試結果，以便我不用手動將數據輸入到 Excel。
3.  **作為合規檢查員**，我想要透過輸入關鍵字（如產品型號）快速找到三年前的測試報告，以便因應外部稽核。
4.  **作為數據輸入員**，我想要在一個介面同時看到 PDF 原件和 AI 辨識結果，以便我能快速修正辨識錯誤並確保數據正確。
5.  **作為生產主管**，我想要閱讀 AI 生成的簡短摘要而非整份報告，以便我在 1 分鐘內掌握測試結論。
6.  **作為管理員**，我想要管理團隊成員的存取權限，以便保護敏感的測試數據不被隨意修改。
7.  **作為品質工程師**，我想要選取特定指標的折線圖，以便分析產品是否存在週期性的失效趨勢。
8.  **作為系統用戶**，我想要將報告標記為不同的標籤（如： urgent, re-test），以便進行分類後續追蹤。

## 6. 非 MVP 範圍 (未來考量)

*   **手機 App 開發**：MVP 階段僅限 Web 端使用。
*   **手寫辨識 (ICR)**：僅處理數位生成的 PDF，不處理手寫掃描件。
*   **自動預警功能**：當趨勢接近臨界值時自動發送 Email/Slack 通知。
*   **第三方系統整合**：如直接將數據推送至 SAP 或 Oracle ERP。
*   **自定義報表生成**：允許用戶自定義輸出格式並導出為 Excel/PDF。
*   **多國語言介面**：MVP 僅提供繁體中文。
*   **AI 自動學習**：用戶修正後的數據自動回傳訓練模型以優化下次辨識。
*   **大檔案批次處理**：超過 500MB 或 1000 頁的超大型檔案處理。
*   **進階統計模型**：如 CPK (製程能力指數) 自動計算。
*   **區塊鏈存證**：對報告進行雜湊值上鏈，確保數據不被篡改。

## 7. 技術要求

### 7.1 後端框架
*   **推薦技術**：Node.js (NestJS 框架)
*   **理由**：NestJS 提供了強大的架構支持，適合處理異步的 AI API 調用與檔案上傳流，且與前端 React 共享 TypeScript 語言生態。

### 7.2 資料庫
*   **推薦技術**：PostgreSQL
*   **理由**：穩定且強大的關聯式資料庫，適合處理結構化測試數據，且對於時序數據的查詢性能優異。

### 7.3 資料庫 ORM
*   **推薦技術**：Prisma
*   **理由**：強型別定義與開發者友好的語法，能大幅提升數據操作的安全與開發效率。

### 7.4 前端技術
*   **推薦技術**：React.js + Tailwind CSS + Ant Design
*   **理由**：React 生態豐富；Ant Design 提供成熟的後台管理介面組件（如表格、圖表、上傳組件），適合快速建構 B2B 系統。

### 7.5 其他整合
*   **AI/OCR 服務**：Azure Form Recognizer 或 OpenAI GPT-4o (Vision) 用於處理複雜表格擷取。
*   **檔案儲存**：AWS S3 或 Google Cloud Storage 用於儲存原始 PDF 文件。

### 7.6 部署環境
*   **建議方式**：Docker 容器化部署，運行於 AWS ECS 或 Vercel (前端) + Render (後端)。

### 7.7 版本控制
*   **建議使用**：Git (GitLab 或 GitHub)，採 Git Flow 分支管理策略。

## 8. 成功指標

*   **數據錄入效率提升 70%**：用戶完成一份報告入庫的平均時間從 10 分鐘縮減至 3 分鐘以內。
*   **數據準確率達 100%**：經過人機協作覆核後，存入資料庫的數值無誤。
*   **檢索速度**：跨 10,000 份報告的查詢請求應在 2 秒內回應。
*   **系統可用性**：系統運行時間 (Uptime) 保持在 99.9% 以上。
*   **用戶滿意度 (CSAT)**：內部測試用戶對數位化查詢功能滿意度評分達 4.5/5。

## 9. 假設與約束

### 9.1 假設
*   用戶上傳的 PDF 報告必須是數位生成的，而非透過低畫質相機拍攝的。
*   實驗室提供的標準模板在短期內不會發生結構性的劇烈更動。
*   用戶端的網路頻寬足以支撐 PDF 檔案（平均 2-5MB）的上傳需求。
*   AI 服務（如 OpenAI/Azure）的 API 回應延遲在可接受範圍內（<15秒）。

### 9.2 約束
*   **法律合規**：系統必須符合企業內部對測試數據的隱私保護政策。
*   **預算限制**：MVP 開發需在既定的開發週期（例如 3 個月）內完成。
*   **技術限制**：AI 對於極度複雜或巢狀表格的辨識率可能受限，需依賴人工修正。
